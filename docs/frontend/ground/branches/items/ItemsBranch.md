# [Список всех предметов (ItemsBranch.ts)](../../../../../Frontend/Web/ground/branches/items/ItemsBranch.ts)

## Назначение
Этот файл представляет собой масштабный компонент, отвечающий за отображение и фильтрацию **Списка всех предметов** в **Backpack Brawl**. Он предоставляет пользователю интерфейс для изучения базы данных игры, включая поиск, фильтрацию по характеристикам и бесконечную прокрутку.

Именно этот файл является главной целью для будущего «Распила монолита», поэтому его документация должна быть максимально детальной.

*   **Ядро**: Наследуется от [Базового класса ветки](../../roots/Branch.md).

## Подробное описание функций и методов

### 1. Рендеринг шаблона (`getHtml`)
Формирует основной каркас страницы. 
*   **Сложность**: Содержит множество выпадающих списков (`dropdown-filter`) для категорий: Типы, Редкости, Герои, Разблокировка, Баффы, Дебаффы, Статы.
*   **ID элементов**: Каждому фильтру назначен уникальный ID (например, `#filterTypes`), который используется позже для делегирования событий.

### 2. Инициализация (`init`)
Жизненный цикл начинается здесь:
*   `this.restoreState()`: Восстановление всех фильтров из `sessionStorage`.
*   Настройка **Debounce** для поиска: задержка в 200мс предотвращает лаги интерфейса при быстром наборе.
*   `initDropdownFilters()`: Настраивает логику открытия/закрытия меню фильтров с анимацией стрелочек.

### 3. Подготовка индекса (`prepareItems`) — **КЛЮЧЕВОЙ ЭТАП**
Поскольку предметов в **Backpack Brawl** очень много, поиск «на лету» по сырым объектам был бы медленным. Этот метод создает «плоский» поисковый индекс `PreparedItem`.
*   **searchText**: Объединяет все текстовые поля предмета в одну нормализованную строку.
*   **strictText**: Создает строку тегов для быстрого вхождения (например, если в описании есть "Critical Chance", в индекс добавится токен `critchance`).

### 4. Парсинг запроса (`parseSearchInput`)
Разделяет строку пользователя на две части:
1.  **Строгие теги**: Всё, что в `[квадратных скобках]`.
2.  **Обычный текст**: Всё остальное.
*   *Пример*: Запрос `ядовитый [Weapon]` будет разделен на текстовый поиск по "ядовитый" и жесткий фильтр по тегу "Weapon".

### 5. Движок фильтрации (`applyFilters`) — **СЕРДЦЕ МОНОЛИТА**
Самый сложный метод, работающий по принципу конвейера (Pipeline):
1.  **Fuse.js + Семантика**: 
    *   Если есть текстовый запрос, он сначала расширяется синонимами через [Сервис семантики](../../utils/SearchTermService.md).
    *   Затем результат прогоняется через Fuse.js.
    *   **Внимание**: Оригинальные слова имеют больший приоритет (меньший `score`), чем синонимы.
2.  **Логика строгих тегов**:
    *   Реализует **AND** между группами скобок и **OR** внутри группы. 
    *   Пример: `[Poison] [Weapon Bow]` найдет предметы, у которых ОБЯЗАТЕЛЬНО есть яд И (либо меч, либо лук).
3.  **Мультиселекты**: Наложение фильтров по героям, редкостям и т.д.

### 6. Сортировка (`sortAndRender` и `compareSearchResults`)
*   Если есть поисковый запрос: Сортировка идет по релевантности (наилучшее совпадение выше).
*   Если поиска нет: Используется [Веса редкостей](../profile/_profile/utils/rarity-weights.md) (Unique выше Common) или алфавитный порядок.
*   **Сохранение порядка**: Массив имен отфильтрованных предметов сохраняется в `filteredItemsOrder`, чтобы на детальной странице предмета работали кнопки "Назад/Вперед" в правильном порядке.

### 7. Бесконечный скролл (`appendNextItemsBatch`)
*   Использует `IntersectionObserver` для слежения за невидимым элементом `#itemsScrollSentinel` внизу страницы.
*   **Batching**: Отрисовывает по 80 предметов за раз, что позволяет мгновенно открывать страницу даже при 1000+ предметов.
*   **Предзагрузка**: При наведении курсора (`pointerenter`) на карточку вызывается [Предзагрузка изображений](../../utils/ItemPreviewPrefetchService.md).

## AI-контекст
*   **Монолитность**: В этом файле смешана логика данных (Fuse.js) и логика UI (DOM-манипуляции). При рефакторинге (п. №1 плана) поисковый движок стоит вынести в отдельный модуль поиска, а отрисовку — в отдельный рендерер (таких классов сейчас НЕ существует — это предложение по декомпозиции монолита).
*   **Память**: Помните о `cleanupFns`. Поскольку бранч подписывается на множество событий (scroll, input, clicks), метод `destroy` должен вызываться всегда.
*   **Кеширование Fuse**: Экземпляр Fuse кешируется глобально (`cachedFuse`), чтобы не пересоздавать индекс при переходах между страницами.
