# [Менеджер навигации по предметам (ItemNavigationManager.ts)](../../../../../../Frontend/Web/ground/branches/itemDetail/_itemDetail/managers/ItemNavigationManager.ts)

## Назначение
Этот менеджер позволяет пользователю переключаться между предметами (кнопки "Назад" и "Вперед") без возврата к общему списку. Он интеллектуально определяет порядок предметов в зависимости от того, откуда пришел пользователь: из общего справочника (Список предметов) или из конкретного профиля игрока.

## Связи (Dependencies)
*   [Типы деталей предмета](../utils/item-detail-types.md): Использует интерфейс `NavigationState`.
*   [SlugService](../../../../../utils/SlugService.md): Используется для точного сопоставления имен предметов в URL и в списке.

---

## Подробное описание методов

### `calculate(itemName: string): NavigationState` — **КРИТИЧЕСКАЯ ФУНКЦИЯ**
*   **Задача**: Найти предыдущий и следующий предметы относительно текущего.
*   **Алгоритм**:
    1.  Определяет контекст: `isProfile` (из Профиля) или `isList` (из Списка всех предметов).
    2.  Читает массив порядка из `sessionStorage` (`profileItemsList` или `filteredItemsOrder`).
    3.  **Слагирование**: Сравнивает имена предметов через `SlugService.toSlug`. Это решает проблему предметов с апострофами (например, "Fool's Gold"), которые иначе могли бы не найтись в JSON.
    4.  Возвращает объект с именами соседей.

### `attachProfileNavListeners(container)`
*   **Задача**: Передавать состояние предмета при клике на кнопки навигации.
*   **Логика**: Если пользователь находится внутри профиля, при переключении на следующий предмет нужно передать не только его статику, но и данные игрока (уровень, количество карт). Метод вешает слушатель на контейнер и подкладывает `playerItem` в объект состояния навигации (`_stateData`).

### `destroy()`
Удаляет слушатели событий с контейнера для предотвращения утечек памяти.

## AI-контекст
*   **Гигиена данных**: Менеджер опирается на `sessionStorage`. Если пользователь откроет две вкладки с разными профилями, может возникнуть конфликт порядка. Это осознанный компромисс ради производительности.
*   **Безопасность**: Весь парсинг JSON из хранилища обернут в `try/catch`, чтобы битый кеш не блокировал работу интерфейса.
