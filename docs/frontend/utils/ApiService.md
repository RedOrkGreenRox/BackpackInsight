# [Frontend/Web/ground/utils/ApiService.ts](../../../Frontend/Web/ground/utils/ApiService.ts)

## Назначение
Главный коммуникационный узел фронтенда. Этот класс инкапсулирует все HTTP-запросы к бэкенду, реализует логику повторных попыток (retries), обработку специфических ошибок **Backpack Brawl** и управление сетевым трафиком.

## Связи (Dependencies)
*   **Бэкенд**: Работает напрямую с эндпоинтами API, описанными в `[api.py](../../backend/playerdata/api.md)`.
*   `[LoadingStates.ts](./LoadingStates.md)`: Часто используется в связке для индикации прогресса запроса.
*   `[global.d.ts](../types/global.md)`: Используется для строгой типизации возвращаемых данных (например, интерфейсов `Profile` и `ItemDefinition`).

## Подробное описание функций и методов

### Перечисления и интерфейсы
*   `ErrorType`: Перечисление типов ошибок (`NETWORK`, `VALIDATION`, `PERMISSION`, `SERVER`).
*   `ErrorInfo`: Интерфейс, описывающий структуру ошибки, включая флаг `retryable` (можно ли повторить запрос).

### Приватные статические методы
*   `getErrorInfo(status: number): ErrorInfo`: 
    *   Анализирует HTTP-код ответа.
    *   Коды `400` и `413` помечаются как `VALIDATION` (нельзя повторять).
    *   Код `429` (Too Many Requests) и `5xx` (ошибки сервера) помечаются как `retryable: true`.
    *   Специальная обработка кода `503` (Service Unavailable), который часто возвращается Edge-функциями Cloudflare, если бэкенд на VPS временно недоступен.
*   `retryWithBackoff<T>(operation, maxRetries)`: 
    *   Реализует алгоритм **экспоненциального бэк-оффа** (exponential backoff). 
    *   Если запрос не удался, следующая попытка произойдет через 1с, затем через 2с, затем через 4с. Это снижает нагрузку на сервер и дает ему время «подняться».
*   `executeWithRetry<T>(operation)`:
    *   Вспомогательная обертка, которая вызывает `retryWithBackoff` и корректно разворачивает сообщения об ошибках, убирая технические префиксы.

### Публичные статические методы
*   `getProfile(jsonData: any): Promise<any>`: 
    *   Отправляет POST-запрос на `/api/profile`.
    *   Содержит логику автоматического повтора при сбоях сети.
    *   Возвращает распарсенный JSON профиля.
*   `getItems(): Promise<any>`: 
    *   Отправляет GET-запрос на `/api/items`.
    *   Использует тот же механизм `executeWithRetry` для стабильного получения списка всех предметов игры.

## AI-контекст
*   **Устойчивость (Resilience)**: При написании кода, вызывающего API, помните, что `ApiService` уже умеет сам переподключаться. Не нужно писать циклы `for` для повторных запросов в компонентах.
*   **Обработка ошибок**: Метод `getProfile` выбросит исключение с ключом локализации (например, `'error_file_too_large'`), которое нужно передать в `[i18n.md](../ground/localization/i18n.md)` для отображения пользователю.
*   **Типизация**: При расширении API всегда добавляйте новые методы сюда, соблюдая паттерн `operation` + `executeWithRetry`.
