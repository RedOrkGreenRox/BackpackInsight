# [Индексы и Таймстампы (0002_profile_indexes_and_timestamps.py)](../../../../../Backend/DB/migrations/versions/0002_profile_indexes_and_timestamps.py)

## Назначение
Вторая ревизия БД, направленная на оптимизацию производительности поиска и внедрение политики хранения данных (Data Retention).

---

## Ключевые изменения

### 1. Жизненный цикл профиля
*   Добавлены колонки `created_at` и `updated_at` в таблицу `profile`.
*   Это позволяет реализовать логику удаления старых профилей (TTL), описанную в [api.py](../PlayerData/api.md).

### 2. Уникальность игроков
*   Внедрен частичный уникальный индекс `uix_profile_user_id`.
*   Он гарантирует, что один `user_id` (игрок) имеет только один актуальный снимок профиля в базе данных.
*   **Миграционная чистка**: Перед созданием индекса скрипт автоматически удаляет дубликаты, оставляя только самые свежие записи для каждого игрока.

### 3. Поиск по предметам (PostgreSQL)
*   **`pg_trgm`**: Активируется расширение для триграммного поиска.
*   **GIN Индексы**: Созданы полнотекстовые индексы на колонки `name` и `item_id` в таблице `itemdefinition`. Это ускоряет нечеткий поиск в сотни раз по сравнению с обычным `LIKE`.

---

## Учет специфики БД
Миграция содержит проверки диалекта (`_dialect()`). GIN-индексы и расширения создаются **только** для PostgreSQL. На SQLite эти шаги пропускаются без ошибок.

## AI-контекст
*   Эта миграция делает схему БД самодостаточной: теперь индексы поиска существуют сразу после развертывания, и приложению не нужно пытаться создавать их программно при каждом старте.
*   Логика очистки дублей (`_dedupe_profiles_by_user_id`) учитывает отсутствие `ON DELETE CASCADE` и сначала чистит связанные таблицы `item` и `hero`.
