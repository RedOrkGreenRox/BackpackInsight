# [Модель профиля (Profile.py)](../../../../Backend/PlayerData/models/Profile.py)

## Назначение
Главная модель данных бэкенда. Представляет собой полный снимок аккаунта игрока **Backpack Brawl**. Профиль агрегирует всех героев, предметы и техническую информацию.

## Связи (Dependencies)
*   [Герои](Hero.md): Отношение `Relationship` (один к многим).
*   [Предметы](Item.md): Отношение `Relationship` (один к многим).
*   [Фабрика профилей](../services/ProfileFactory.md): Используется для создания объекта из JSON.

---

## Подробное описание функций и методов

### Поля БД
*   **Индексы**: Поля `nickname`, `trophies`, `level` и `user_id` проиндексированы для мгновенного выполнения аналитических запросов и поиска.
*   **Уникальность `user_id`**: Частичный уникальный индекс (`0002`-миграция, `WHERE user_id IS NOT NULL`) гарантирует, что один игрок = одна строка. Это основа upsert-логики в [`api.py`](../api.md): повторная загрузка профиля обновляет существующую запись.
*   **Таймстампы**: `created_at` / `updated_at` (миграция `0002`) — поддержка upsert-семантики и политики хранения/TTL (`cleanup_old_profiles`).
*   **JSON Хранилище**: Использует поля `technical_info_data` и `game_info_data` для хранения "сырых" данных. Это позволяет сохранять полную информацию из логов игры, даже если структура приложения еще не умеет обрабатывать все поля.

### `classmethod from_json(json_data)`
*   **Задача**: Точка входа для создания профиля.
*   **Логика**: Делегирует работу [Фабрике профилей](../services/ProfileFactory.md). Это защищает модель от раздувания кодом парсинга и решает проблему циклических импортов.

### Свойства (Properties)
*   `technical_information` / `game_information`: Обертки, которые на лету превращают JSON-данные из БД в типизированные объекты `TechnicalInfo` и `GameInfo`. Это обеспечивает автодополнение в коде и валидацию данных.

### `to_frontend_view()`
*   **Задача**: Конвертация в формат для [Главного фронтенд-сервиса](../../../frontend/ground/utils/ApiService.md).
*   **Агрегация**: Собирает данные из всех связанных моделей и вычисляет сводные показатели (общее количество героев, предметов, текущая зона арен).

## AI-контекст
*   **Уникальность**: Поле `user_id` соответствует игровому `UID`. Это главный ключ для идентификации повторных загрузок одного и того же игрока.
*   **Архитектура**: Профиль спроектирован так, чтобы его можно было легко сериализовать в Jinja2 шаблоны или JSON-ответы FastAPI.
