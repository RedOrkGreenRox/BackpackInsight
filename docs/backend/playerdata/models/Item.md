# [Модели Предметов (Item.py)](../../../../Backend/PlayerData/models/Item.py)

## Назначение
Этот файл содержит две ключевые модели, разделяющие статические данные игры и динамические данные игрока.

---

## 1. ItemDefinition (База знаний)
Хранит общую информацию о предмете, которая одинакова для всех игроков.
*   **`item_id`**: Уникальный строковый ID из ресурсов игры.
*   **`recipes`**: Список комбинаций (ингредиентов), из которых можно собрать предмет.
*   **`combat_stats_data`**: JSON-блок с параметрами урона, скорости и точности.
*   **`item_shape`**: Геометрия предмета (сколько клеток занимает в рюкзаке).

## 2. Item (Инвентарь игрока)
Описывает конкретный экземпляр предмета у игрока.
*   **`level` / `cards`**: Текущий прогресс прокачки предмета.
*   **`definition_id`**: Ссылка на статическое определение.
*   **Proxy Properties**: Использует декоратор `@definition_proxy_property` для прозрачного доступа к полям определения (например, `item.name` фактически лезет в привязанный `item.definition.name`).

---

## Ключевая логика
*   **`cards_need`**: Рассчитывает количество карт, необходимых для следующего уровня, на основе редкости предмета.
*   **`total_xp`**: Вычисляет суммарный опыт предмета, который вносит вклад в общий уровень профиля игрока.
*   **`tags`**: Динамически определяет свойства предмета (Upgradable, Craftable), используемые для фильтрации на фронтенде.

## Связи (Dependencies)
*   **БД**: Связь "Многие-к-одному" с [Profile](Profile.md).
*   **Утилиты**: Использует [utils.py](../utils.md) для работы прокси-свойств.

## AI-контекст
*   Использование `sa_column=Column(JSON)` позволяет хранить сложные вложенные структуры данных игры без создания сотен мелких таблиц.
*   Метод `_get_definition` имеет встроенный фолбэк на кэш `ProfileFactory`, что предотвращает ошибки при работе с отсоединенными от сессии объектами.
