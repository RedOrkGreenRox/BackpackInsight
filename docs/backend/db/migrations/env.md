# [Среда миграций (env.py)](../../../../Backend/DB/migrations/env.py)

## Назначение
`env.py` — это связующее звено между Alembic и кодовой базой приложения. Этот скрипт настраивает контекст выполнения миграций, обеспечивая доступ Alembic к моделям данных и подключению к БД.

---

## Ключевая логика

### 1. Регистрация моделей
Скрипт выполняет импорт:
```python
import Backend.PlayerData.models
```
Это необходимо для того, чтобы все классы (наследуемые от `SQLModel`) зарегистрировали свои метаданные в `SQLModel.metadata`. Без этого `autogenerate` не сможет увидеть изменения в моделях.

### 2. Использование живого Engine
Вместо того чтобы брать URL из конфига `alembic.ini`, скрипт импортирует настроенный `engine` из [database.py](../database.md). Это гарантирует, что миграции всегда применяются к той базе, с которой работает приложение (включая автоматическое переключение на SQLite).

### 3. Режимы работы
*   **Offline**: Генерация SQL-скриптов без реального подключения к БД.
*   **Online**: Применение изменений напрямую к базе данных в транзакции. Настроен параметр `render_as_batch` для корректной работы с SQLite (которая не поддерживает `ALTER TABLE` напрямую).

## AI-контекст
*   Параметр `compare_type=True` позволяет Alembic отслеживать не только создание таблиц, но и изменение типов колонок (например, с `Integer` на `Float`).
*   Если вы добавили новую модель (таблицу), но `alembic revision --autogenerate` её не видит — проверьте, импортирована ли она в `Backend/PlayerData/models/__init__.py`.
