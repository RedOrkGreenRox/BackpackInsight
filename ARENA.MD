# ARENA.MD — Правила работы агента в репозитории BackpackInsight

> **Мета-инструкция для агента:** в начале и в конце каждого сообщения в этом диалоге обязательно писать: «Агент следует правилам ARENA.MD.»

---

## 1. Сетевая документация (Network Documentation)

Это главный закон проекта. BackpackInsight использует **атомарную сетевую документацию** — документация является «живым цифровым двойником» кодовой базы и максимально оптимизирована для понимания ИИ и человеком.

### 1.1. Документация прежде кода
1. При любых изменениях сначала обновляется или создаётся соответствующий `.md` файл в `docs/`.
2. Документация — это техническое задание и архитектурный план для будущего кода.
3. Только после того, как сеть документации обновлена, приступаем к изменению кодовой базы (`.ts`, `.py`, `.scss`, `.sql` и т.д.).
4. После завершения кодинга документация редактируется ещё раз, чтобы устранить любые расхождения. В сети должна остаться **абсолютная истина**.

### 1.2. Принцип 1:1
- Каждый файл исходного кода (`.ts`, `.py`, `.scss`, `.js`, `.sql`, `.mako`) должен иметь зеркальный `.md` файл в `docs/`.
- Исключения:
  - Сгенерированные данные (JSON-справочники предметов).
  - Бинарные ассеты (изображения, шрифты).
  - Конфиги, уже описанные централизованно (`.gitignore`, `.dockerignore`).
  - **Meta-файлы правил (`ARENA.MD`)** — они сами являются документацией и не требуют зеркального `.md`.
- Для новых файлов кода сразу создаётся документ-заглушка, который затем детализируется.

### 1.3. Ссылочная целостность
- Вместо повторного описания логики используются ссылки на документацию зависимых файлов.
- Формат ссылки: `[Название компонента](путь_к_документации.md)`.
- Пример: `[Фабрика профилей](backend/playerdata/services/ProfileFactory.md)`.
- В каждом документе файлового кода в первом заголовке (H1) должна быть ссылка на исходный файл.

### 1.4. Структура центральных узлов
- `README.md` — корневой узел, точка входа для всех агентов и разработчиков.
- `docs/structure.md` — центральная карта проекта, генерируется автоматически.
- `docs/backend/index.md` — точка входа бэкенда.
- `docs/frontend/index.md` — точка входа фронтенда.
- `docs/data/index.md` — точка входа данных и механик.

### 1.5. Автоматическая генерация и проверка
- После любого изменения дерева файлов запускается:
  ```bash
  python scripts/generate_structure.py
  ```
  Это обновляет `docs/structure.md`.
- После рефакторинга или создания новых файлов запускается:
  ```bash
  python scripts/check_docs.py
  ```
  Скрипт проверяет:
  - `[LINKS]` — битые ссылки.
  - `[TREE]` — нет «сирот» (документов, недостижимых из README/structure).
  - `[MIRROR]` — соблюдение 1:1 для исходников и документов.
  - `[STRUCT]` — H1-ссылка на исходник и ключевые секции.
  - `[COMPLETE]` — покрытие символов (классов, функций, селекторов) ≥ 50%.
  - `[TRUTH]` — детектор лжи: символы, которых документ утверждает, но которых нет в коде.

### 1.6. Подпись документации
- Каждый документ в `docs/` должен заканчиваться подписью:
  ```markdown
  ---
  > 📌 **Подпись документации:** …
  ```
- Для ручных документов указывается дата аудита.
- Для автогенерированных — источник генерации.

### 1.7. Документация описывает текущее состояние, а не изменение
- Каждый документ должен описывать **актуальное состояние** кода, а не планы, TODO или будущие изменения.
- Если функционал планируется, но ещё не реализован, это должно быть явно обозначено в отдельном разделе «Планируется» или «В разработке».
- Основной текст документации не должен содержать глаголов в форме будущего времени («будем делать», «планируем», «скоро») в описании текущего поведения.
- После любого рефакторинга или изменения логики документация должна быть приведена в соответствие с кодом **до** коммита.

### 1.8. Поведение агента при расхождении
- **Если код не совпадает с описанием в документации — это критический баг.**
- Агент обязан либо исправить код, либо обновить документацию так, чтобы достичь истины.
- Нельзя оставлять неактуальные документы или «заглушки» без содержательного описания.

## 2. Размер файлов и модульность

Проект поддерживает строгие лимиты на размер файлов, чтобы сохранить поддерживаемость и тестируемость.

### 2.1. Лимиты строк (действуют на все файлы проекта)
Правило распространяется на все текстовые файлы: `.ts`, `.py`, `.scss`, `.js`, `.sql`, `.mako`, `.md` и другие.

| Уровень | Максимум | Когда допустимо |
|---------|----------|-----------------|
| **Желательно** | 80 строк | Всегда, особенно для чистых функций/утилит и документов. |
| **Пограничный** | 150 строк | Для комплексных менеджеров, рендереров или подробных документов, если разделение усложнит читаемость. |
| **Крайний** | 250 строк | Только в исключительных случаях, с явным обоснованием. |

### 2.2. Правило деления при превышении пограничного лимита
- Если файл превышает **150 строк**, агент обязан **попытаться разделить его на модули**.
- Деление должно опираться на **разделение ответственности** (Single Responsibility Principle), а не на механическое разбиение по строкам.

### 2.3. Критерии разделения ответственности
- **UI-логика отдельно от бизнес-логики.**
  - Рендереры отвечают только за генерацию HTML.
  - Менеджеры отвечают за состояние, события, API-вызовы.
  - Сервисы отвечают за чистые функции (поиск, сортировка, валидация, кеширование).
- **Платформенно-специфичный код отдельно от абстрактной логики.**
  - DOM-манипуляции, браузерные API, `fetch` — изолированы.
- **Состояние отдельно от действий.**
  - `*StateManager` — хранит и восстанавливает state.
  - `*Manager` — оркестрирует действия.
- **Повторно используемый код выносится в `utils/` или `services/`.**

### 2.4. Примеры корректного деления
- Монолит `ItemsManager.ts` → разделить на:
  - `ItemsFilterController` — фильтры и поиск.
  - `ItemsSortController` — сортировка.
  - `ItemsUrlController` — синхронизация URL.
  - `ItemsUiController` — чипсы, панель, рендер формулы.
  - `ItemsStateManager` — сохранение/восстановление состояния.

### 2.5. Исключения
- Автогенерированные файлы (миграции Alembic, JSON-справочники).
- Центральные hub-файлы (`docs/structure.md`, `README.md`), если их размер обусловлен структурой.
- Большие конфигурационные файлы, если они не содержат логику.
- **Meta-файлы правил (`ARENA.MD`)** — они содержат нормативы и не подлежат разделению.

### 2.6. Отслеживание нарушений
- `docs/oversized_frontend_files.md` — журнал файлов, превышающих лимиты.
- После рефакторинга агент обязан обновить этот журнал.

## 3. Соответствие структуре эталонов (Frontend)

Во фронтенде существуют **эталонные страницы**, которые демонстрируют правильную гранулярную структуру. Эталонные страницы — это те, где **нет файлов больше 150 строк** и структура разбита на мелкие, специализированные модули.

### 3.1. Общая структура страницы (Branch)
Каждая страница фронтенда должна следовать паттерну `Branch`:

```
ground/branches/{pageName}/
├── {PageName}Branch.ts          # Класс страницы, наследует Branch
├── {pageName}.scss              # Стили страницы
└── _{pageName}/                 # Внутренние модули страницы
    ├── index.ts                 # Реэкспорт модулей
    ├── components/              # UI-компоненты / рендереры
    ├── managers/                # Бизнес-логика и оркестраторы
    ├── services/                # Чистые функции и сервисы (опционально)
    ├── utils/                   # Типы и хелперы (опционально)
    └── {feature}/               # Подмодули по фичам (например, header/, heroes/)
```

### 3.2. Эталонные страницы
- **Главная страница (`main`)**: компактные менеджеры (`MainManager`, `SubmitManager`, `DraftManager`, `ValidationManager`), разделённые обработчики ввода.
- **Детализация предмета (`itemDetail`)**: `ItemDetailManager`, `ItemDataLoader`, `ItemNavigationManager`, `ItemSEOManager`, `ItemDetailRenderer`.
- **Ошибка 404 (`404`)**: мелкие компоненты (`background`, `button`, `container`, `text`, `title`, `navigation`).

### 3.3. Антиэталоны (требующие рефакторинга)
- **`ItemsManager.ts` (402 строки)** — god object, превышает крайний лимит.
- **`ProfileManager.ts` (327 строк)** — god object, превышает пограничный лимит.
- Эти файлы должны быть декомпозированы по примеру эталонных страниц.

### 3.4. Правила гранулярности
1. **Один класс — одна ответственность.**
   - `*Renderer` — только генерация HTML.
   - `*Manager` — только инициализация, события, координация.
   - `*Service` — только чистые функции.
2. **Один SCSS-файл — один визуальный компонент или один аспект.**
   - Не смешивать стили кнопок, сеток и анимаций в одном файле.
3. **Обработчики событий изолированы.**
   - `ClipboardHandler`, `DragDropHandler`, `FileHandler`, `UIHandler` — пример правильного разделения.
4. **Валидация и отображение ошибок разделены.**
   - `JsonValidator` — чистая логика валидации.
   - `ErrorDisplayManager` / `ValidationManager` — отображение.

### 3.5. Проверка перед завершением работы
- После рефакторинга страницы запустить:
  ```bash
  cd Frontend/Web && npm run build && npx vitest run
  ```
- Убедиться, что ни один новый файл не превышает 150 строк без обоснования.
- Обновить `docs/oversized_frontend_files.md`.

## 4. Общие правила поведения агента

### 4.1. Порядок действий
1. Прочитать `README.md` и `ARENA.MD`.
2. Перейти к `docs/structure.md` и нужному разделу.
3. Обновить документацию.
4. Изменить код.
5. Обновить документацию ещё раз.
6. Запустить тесты и `check_docs.py` / `generate_structure.py`.
7. Обновить `docs/oversized_frontend_files.md` при необходимости.

### 4.2. Запрещено
- Оставлять код без документации.
- Создавать файлы больше 150 строк без попытки разделения.
- Использовать `@ts-ignore` без крайней необходимости; предпочитать `@ts-expect-error` или исправление типов.
- Копировать логику вместо ссылок на документацию.
- Игнорировать предупреждения линтера документации.

### 4.3. Обязательно
- Все новые маршруты, модели, сервисы покрывать тестами.
- Соблюдать `.gitattributes` (LF-only для текстовых файлов).
- Не коммитить `.env` и секреты.
- После изменения дерева файлов запускать `generate_structure.py`.
- После рефакторинга запускать `check_docs.py`.
